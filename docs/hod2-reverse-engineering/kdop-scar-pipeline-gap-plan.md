# KDOP & SCAR Pipeline Gap — Implementation Plan

## Executive Summary

Our HOD 2.0 creation pipeline (`generate_v2_from_model`) has two critical gaps vs the OBJ→DAEnerys→HODOR reference pipeline:

1. **KDOP** (K-Discrete Oriented Polytope) collision trees are not generated from scratch. They are only preserved from original files. From-scratch DAE imports get COLD (HOD 1.0 format) instead.
2. **SCAR** (battle scar) chunks are not generated from scratch. They are only preserved from original files.

Both are generated by HODOR for every HOD 2.0 file. Missing KDOP likely causes collision failure or crashes.

---

## Gap 1: KDOP Generation (CRITICAL)

### Current State
- `hod.rs:1240` — KDOP is parsed as raw bytes into `preserved_chunks`
- `hod.rs:5391-5410` — KDOP is written back verbatim during generation
- `hod.rs:5413-5487` — COLD is generated for from-scratch imports (wrong chunk type)
- `hod2-creation-specification.md:326` — Marked "⏳ NOT YET IMPLEMENTED"

### What HODOR Does
- HODOR computes a KDOP collision tree from the mesh geometry
- KDOP is ~1588 bytes for typical models
- It is a K-discrete oriented polytope — a bounding volume hierarchy used for collision detection
- HODOR's DTRM always contains KDOP for HOD 2.0 files

### What Needs to Happen

#### Step 1: Reverse-engineer KDOP binary format
- Dump KDOP from multiple vanilla HOD files (pebble_0, ter_elysium, ter_zephyrus, ter_centaur)
- Analyze byte structure — is it a fixed-size tree? What are the fields?
- Use `dump_kdop.rs` as starting point
- Compare KDOP across models of different complexity to identify variable vs fixed fields
- Document in `hod2_reverse_engineering_knowledge_base.md`

#### Step 2: Implement KDOP generator
- Create `generate_kdop(mesh_vertices: &[HODVertex], mesh_indices: &[u16]) -> Vec<u8>`
- Input: the visible mesh geometry (positions only)
- Output: the raw KDOP chunk data (~1588 bytes)
- Place in `parser/src/kdop.rs` or as a function in `hod.rs`

#### Step 3: Integrate into `generate_v2_from_model`
- Replace COLD generation with KDOP generation for HOD 2.0
- Remove the COLD fallback for v2 files (`hod.rs:5413-5487`)
- Add KDOP as a DTRM child instead of COLD

#### Step 4: Verify
- Run `cargo run --bin verify_lossless` on round-tripped files
- Run `cargo run --bin testing_diff` to compare KDOP structure
- Test in-game if possible

## Gap 2: SCAR Generation (MEDIUM)

### Current State
- `hod.rs:1240` — SCAR parsed as raw bytes into `preserved_chunks`
- `hod.rs:5404` — SCAR written back verbatim
- `analyze_scar.rs`, `analyze_scar2.rs` — Partial reverse engineering underway

### What HODOR Does
- HODOR generates SCAR chunks from mesh geometry
- SCAR contains battle scar vertex/index data for visual damage effects
- SCAR is optional — models work without it, but scars won't render

### What Needs to Happen

#### Step 1: Complete SCAR format reverse engineering
- `analyze_scar.rs` has partially decoded the header: mesh name, parent name, LOD count, BBOX, BSPH
- `analyze_scar2.rs` has identified vertex/index layout after the header
- Finish documenting the full binary layout in `hod2_reverse_engineering_knowledge_base.md`

#### Step 2: Implement SCAR generator
- Create `generate_scar(mesh_vertices, mesh_indices, mesh_name, parent_name) -> Vec<u8>`
- SCAR appears to wrap the visible mesh geometry with bounding info
- Place in `parser/src/scar.rs` or as a function in `hod.rs`

#### Step 3: Integrate into `generate_v2_from_model`
- Add SCAR as a DTRM child when the model has scar-capable geometry
- Only generate if `has_mult_tags` is true (DoScars TAGS present)

#### Step 4: Verify
- Compare generated SCAR against vanilla HOD SCAR chunks
- Test in-game for visual scar rendering

## Gap 3: DAE Texture Mapping (HIGH)

### Current State
- `dae.rs:60-73` — Only resolves diffuse texture from `<library_effects>`
- Non-diffuse slots require DAEnerys `MAT[...]_SHD[...]` naming

### What Needs to Happen
- Extend DAE parser to resolve all texture slots (glow, team, normal, specular)
- Or: document that DAEnerys naming convention is required for full texture mapping
- Consider reading SHADERS.MAP for channel mapping

---

## Gap 4: Tangent Path for Tagged Meshes (MEDIUM)

### Current State
- `compiler.rs:519-532` — `has_mult_tags` path now has `is_already_indexed` check
- For flat triangle lists with tags, tangents are computed via `compute_tangent_space` on deduped vertices
- HODOR accumulates tangents on raw vertices before deduplication

### What Needs to Happen
- Route flat-list `has_mult_tags` parts through `compile_hodor_style_tangent_part` instead of `compute_tangent_space`
- This prevents smoothing across sharp UV seams

1. **Before any changes**: Run `cargo run --bin verify_lossless` and `cargo run --bin test_hodor_replication` to establish baseline

4. **After all changes**: Run full test suite; test in-game with Homeworld Remastered if possible
